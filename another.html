👤 User Administration

SU01 — User Maintenance

What is User Administration?

User Administration involves all tasks related to user maintenance in SAP. The primary transaction code is SU01.

SAP Security ensures users can only access functionality that is part of their job role — balancing security with usability.

SU01 Operations

CreateAdd a new user to the system
ModifyUpdate existing user details
DisplayView user information (read-only)
DeleteRemove a user from the system
CopyClone a user (address & password NOT copied)
Lock / UnlockRestrict or restore user access
Change PasswordReset the user's password

SU01 Create — 11 Tabs

  • Documentation — Description and document for the user
  • Address — Personal info (Last Name is mandatory)
  • Logon Data — User type & initial password (mandatory)
  • SNC — Skip password if Secure Network Communication is active
  • Defaults — Date format, time format, printer, time zone
  • Parameters — Only entered when requested by user
  • Roles — Assign authorizations through roles
  • Profiles — Auto-assigned when roles are assigned (max 312)
  • Groups — Assign user to extra groups
  • Personalization — Do not enter details here
  • Lic. Data — Licensing (Admin, Developers, etc.)

Key Facts

User ID max length12 characters
Max profiles per user312
Mandatory fieldsLast Name + Initial Password
User tables prefixUSR* (e.g., USR02, USR40)
Change docs pathSU01 → Information → Change Documents for Users

Security Types

Internal SecurityManaged by SAP authorization concept — application level
External SecurityManaged by database system — database level

The typical workflow when a user needs SAP access:

🏷️ User Types & Locks

Logon Data — User Types & UFLAG

Five User Types in SAP

1. DialogDefault type. Interactive. Password rules apply. GUI login allowed.
Example: All employees
2. SystemInternal communication. No password rules. NO GUI login.
Example: Background jobs
3. CommunicationExternal system communication. No password rules. NO GUI login.
Example: RFC connections
4. ServiceMultiple dialog logons. No password rules. GUI login allowed.
Example: FFID in GRC
5. ReferenceExtra access for dialog IDs at 312 profile limit. No password. NO GUI login.
Example: Extend access via reference user

Reference User Scenario

Create a Reference User with 2 roles → Assign 1 role to a Service User + assign the Reference User → Service User now has access to 3 roles!

UFLAG Values — User Lock Status

0Not Locked
32Global Lock (CUA)
64Administrator Lock
128Incorrect Logon Lock (Password Lock)
192 (128+64)Password Lock + Administrator Lock
160 (128+32)Password Lock + CUA Lock

Password Parameters Comparison

Dialog✅ Password rules applied
System❌ No password rules
Communication❌ No password rules
Service❌ No password rules
Reference❌ No password required

🔐 Authorization Concepts

Auth Objects, Fields & T-Codes

Authentication vs Authorization

AuthenticationVerifies your identity (User ID + Password). Exists outside SAP system. Common to every user.
AuthorizationControls permissions inside SAP. What you can do after logging in. Varies per user/system.
Security = Authentication + Authorization. Strong auth check: Biometric!

Authorization Structure

The hierarchy of SAP authorization:

Authorization Object └── Auth Field 1 → Value └── Auth Field 2 → Value └── ... (max 10 fields) └── Auth Field 10 → Value Object Class (folder) → holds Authorization Objects
SU21T-code for Authorization Objects maintenance
SU24Find auth objects related to any T-code
SE93Maintain all T-codes (find program & auth object)
SA38Execute ABAP programs directly

SU01 Auth Objects

S_USER_GRPCreate users
S_USER_AGRAssign roles to users
S_USER_PROAssign profiles to users
⚠️ T-codes and auth objects CANNOT be assigned directly to users — they must be placed inside a Role first!

T-Code Explained

A T-code (Transaction Code) is a memorable shortcut to navigate to a program without remembering long program names.

SU01 → Program: SAPMSUU0 SCC5 → Program: SAPMSCC1 Path to run program directly: SA38 → enter program name → Execute

🎭 Roles & Profiles

PFCG — Profile Generator

What is a Role?

A Role is a group of T-codes and their related authorizations bundled together. Roles are assigned to users — NOT T-codes or auth objects directly.

Profile Types

Standard ProfilesCome with SAP installation. Can be assigned directly. SAP recommends NOT using these — they give extra access.
Generated ProfilesCreated by PFCG when roles are built. Always start with "T". Cannot be assigned directly. Auto-assigned when role is assigned.

PFCG Tabs — Single Role

DescriptionLong text with why/when/who created the role
MenuAdd T-codes via the Transactions option
AuthorizationsChange auth data & generate profile
UserList of users — click User Comparison to turn green
PersonalizationDo NOT touch this tab

Role Naming Convention

Format: [Type]:[Business]:[Sub-process]:[Client] Types: X = Single | Y = Composite | Z/P = Derived/Master Max length: 30 characters Example: X:HR:PERSONAL_ADMIN:TATA

Role Tables (AGR*)

AGR_DEFINERole definitions, master/derived info
AGR_AGRSSingle and composite roles
AGR_TCODESRoles and their T-codes
AGR_USERSUsers assigned to roles
AGR_1251Role and authorization values
AGR_1252Role and org values

Single Role

Basic role with T-codes and authorization objects. Created in PFCG → Single Role button.

Composite Role

A group of single roles bundled together for easy assignment. Reduces security admin tasks and simplifies user experience.

Example: Security team needs → Security Admin Role + ABAP Display Role → Bundle as Y:SECURITY_COMPOSITE_ROLE

Tabs: Description, Roles, Menu (click Import Menu), User, Personalization

Master Role (Parent)

Has full access with org level set to * (wildcard). Modified like a single role. Changes propagate to derived roles.

Derived Role (Child)

Single role derived from a master role. Inherits all T-codes and auth objects. Only differs in org field values (e.g., company code, plant).

Classic use case: MNC like Coca-Cola needs India finance users to only process India invoices (company code=100) and USA users for USA invoices (company code=200) — using the same FBV0 T-code.
Master: Company Code = * ├── India Derived: Company Code = 100 ├── USA Derived: Company Code = 200 └── UK Derived: Company Code = 300 No limit on number of derived roles!

Expert Mode — Profile Generation

Delete & RecreateWipes all old data including customization. Fresh load from SU24. Use when starting fresh.
Edit Old StatusKeeps old data + customization. Does NOT pull new T-code auth data. Use for direct auth changes.
Read & MergeKeeps old + customization AND pulls new T-code data. BEST option for role modification!

🚦 Authorization Tab Deep Dive

Traffic Lights, Fields & Statuses

Traffic Light System

🟢 GreenAll authorization fields filled/maintained with values
🟡 YellowAt least one authorization field not maintained or partially unmaintained
🔴 RedUnmaintained organizational levels

Field Types

Normal FieldDoes NOT represent organization. Universal meaning.
Example: Activity 03 = Display (everywhere)
Organizational FieldRepresents organization. Meaning varies by company.
Example: Plant 100 = UK plant at BMW; Plant 100 = USA branch at Walmart
⚠️ Org field values must be maintained centrally from the Organization tab. Normal field values can be maintained directly.

Auth Object Statuses

StandardSAP default values pulled from SU24 for T-codes in menu
MaintainedBlank fields have been filled with values
ChangedSAP default values have been modified (Activity values changed)
ManuallyAuth object inserted directly into role, NOT from SU24

Deleting Auth Objects

Both conditions must be met:

  • Auth object must be deactivated
  • Status must be Manually or Changed

Testing Methods

Positive TestingVerify user CAN execute T-codes assigned in the role
Negative TestingVerify user CANNOT execute T-codes NOT in the role

🔍 Troubleshooting

SU53, SU56, SUIM, ST01, STAUTHTRACE

Common Issues

Cannot loginWrong password, user locked, user expired, incorrect client, password expired
Can login but not execute T-codeRole expired, profile not generated, user comparison not done
Can execute T-code but limited functionsMissing auth values, profile not generated, user comparison not done

SU53 — Authorization Check

Evaluates the last authorization check — shows if it passed or failed, and reveals missing objects/fields.

Run SU53 immediately after getting an authorization error — it captures the last check!

SU56 — User Buffer

Temporary memory created when user logs in, wiped on logout. System checks buffer — NOT database — when T-codes are executed.

Buffer issue fixAsk user to logout and login again, OR reset buffer in SU56
Check bufferSU56 → Authorization Values → User Buffer

Return Codes (ST01 / STAUTHTRACE)

RC = 0✅ Authorization successful
RC = 4Has auth object but wrong values
RC = 8No required auth values in user buffer
RC = 12No access to auth object at all

ST01 vs STAUTHTRACE

ST01System Trace — local server only. Must be on same server as user. Check AL08 / SM51 to verify.
STAUTHTRACEExclusive auth trace — works across ALL application servers. Available in updated SAP versions.

SUIM — User Information System

Powerful audit tool for security administrators.

  • Users by Address Data / Auth Values / Role Assignment
  • Roles by Complex Criteria / User Assignment / Profile Assignment
  • Change Documents for Users, Roles, Profiles
  • Where-Used List for auth objects and fields

🚚 Transport Management

TMS — Dev → QA → PRD

Transport Overview

Transport = Movement of data from one SAP environment to another via Transport Requests (TR).

DEV ──→ QA ──→ PRD (Development) (Quality) (Production)

In QA and PRD: PFCG display access only. All changes made in DEV.

Transport Request Structure

TR Name Format: [SID]K9[5-digit number] Example: W88K900035 Transport Request ├── Task 1 (User: VARUN) → Role 1 └── Task 2 (User: ANKITA) → Role 2 Task = Space reserved for ONE user. Only that user can load data into the task.

TR Types

Workbench TRCarries SAP standard data (ABAP code, T-codes, programs)
Customizing TRCarries customized data (business data, users, roles, transactional data)

T-codes for TR: SE01 SE09 SE10. Tables: E070, E071

Security Team TR Duties

  1. Create Transport Request (SE10 or SE09)
  2. Load role into Transport Request (PFCG)
  3. Release the Transport Request (SE10/SE09)
  4. Intimate Basis Team to transport TR into Quality
⚠️ Before releasing TR, ensure ALL tasks under it are released first!

Role Transport Screen — Important!

When transporting a role via PFCG, NEVER tick:

  • Personalization Data
  • Direct User Assignment

Only keep "Generated Profiles of Single Roles" checked.

🏗️ Implementation Phase

SU25, SU22, SU24 & First Installation

Project Types

ImplementationNo SAP at all → implementing SAP for the first time
SupportSAP already exists → fixing issues (long-term)
UpgradeVersion upgrade
RolloutExtending SAP to other business branches

SU22 → SU24 → PFCG Flow

SU22 (SAP Standard) ──→ SU24 (Customer Custom) ──→ PFCG (Role Auth Data) USOBT/USOBX USOBT_C/USOBX_C When T-codes added in PFCG Menu: Auth data pulled FROM SU24 automatically!
If SU24 is empty → PFCG T-codes won't pull auth data! Use SU25 Step 1 to copy SU22 → SU24.

SU25 Steps

Step 1Initial fill — copies SU22 to SU24 (overwrites, loses customization)
Step 2AAutomatic comparison with SU22 (standard check)
Step 2BModification comparison — choose SAP suggested or custom (SU24)
Step 2DShow changed T-codes (old vs new)
Step 2CRoles to be checked after T-code changes
Step 3Transport customer tables (SU24 changes to other systems)

Key Tables

USOBTSAP standard auth values for T-codes (SU22 data)
USOBXCheck indicators for auth objects per T-code (OK flag)
USOBT_CCustomization + Standard (SU24 data)
USOBX_CStandard + Customization (SU24 data)
