User Administration — SU01

👤
All user-related activities are handled via SU01 — User Maintenance T-code.
User Activities
User Creation User Modification User Deletion User Lock / Unlock User Copy Password Reset
SU01 Tabs — Explained
TabPurpose
AddressFirst name, last name, email, department info
Logon DataUser Type, User Group, Validity Period (Valid From / Valid Thru)
SNCSecure Network Communication — when activated, no password needed to login
DefaultsDate format, decimal notation, time format, printer details — common settings
ParametersOnly entered when requested by user (e.g. default values for fields)

GroupsAssign user to extra groups if they belong to multiple groups
Personalization⚠ Do NOT enter any details here
License DataRelated to user licensing
Change DocsView all changes made to user in a time interval — Path: SU01 → Information → Change Documents for User
User Types — All 5
1️⃣ Dialog — Default User Type ▾
  • Interactive user — humans logging in daily
  • Password parameters ARE applied
  • GUI Login is allowed
  • Example: All employees of the organization
2️⃣ Service — Multiple Dialog Logons ▾
  • Used for multiple dialog logons simultaneously
  • Password parameters are NOT applied
  • GUI Login is allowed
  • Example: FFID in GRC, TEST IDs
3️⃣ System — Internal Communication ▾
  • Password parameters are NOT applied
  • GUI Login NOT allowed
  • Example: Background jobs, Internal RFC
4️⃣ Communication — External Communication ▾
  • Password parameters are NOT applied
  • GUI Login NOT allowed
  • Example: RFC (Remote Function Call)
5️⃣ Reference — Extra Access Provider ▾
  • Used when Dialog user hits the 312-profile limit
  • No password required
  • Cannot login through GUI
  • Provides extra access to Dialog IDs

Types of Profiles
Standard Profiles
  • Come with SAP installation
  • Can be assigned to users directly
  • SAP recommends NEVER assign — gives extra access
Generated Profiles
  • Created through Roles (PFCG)
  • Cannot be assigned to users directly
  • Start with T alphabet (e.g. T:BW:REPORT_EXECUTION)
  • Assign the Role to users instead
🔴
If a profile like 1-87970004 is generated, assign the associated role X:BW:REPORT_EXECUTION to the user — not the profile directly.

  • Maintained Blank fields filled with values

Deleting Auth Objects from Role (Prerequisites)
  • Auth Object must be Deactivated
  • Auth Object status must be MANUALLY or CHANGED
Profile Generation Modes
Manual Mode
Go to Change Authorization Data and click Generate Profile.
Expert Mode
Used during Role Modification ONLY (not role creation). Has 3 sub-options.
Expert Mode — 3 Options
🗑 Delete and Recreate Profile and Authorizations ▾
  • Deletes entire old data including customization
  • Freshly loads auth data from SU24 for all T-codes in Menu
  • Use when customization needs to be deleted
✏ Edit Old Status ▾
  • Keeps old data as-is (including customization)
  • Does NOT pull auth data for newly added T-codes
  • Use when auth data needs direct modification without menu changes
🔀 Read Old Status and Merge with New Data ✓ RECOMMENDED on Menu Change ▾
  • Keeps old data as-is (including customization)
  • Pulls auth data for ALL T-codes in Menu (including new ones)
  • Merges old data with new data wherever possible
  • Best option when role menu has changed

06 — DERIVED ROLES

Master Role → Derived Roles

🌿
A Derived Role is a single role derived from another single role (Master Role). Only Organizational Level values differ between the master and derived roles. All menu/auth structure is inherited.
When to Use Derived Roles?
🌐
When the client is an MNC (Multinational Company) or has multiple business units: Coke, BMW, NESTLE, TATA, DUPONT, etc. — same T-codes but different org values (e.g., different Company Codes or Plants).
Example: India vs USA Finance
Master Role
FBVO / F_FICO_BUK
Comp Code: *
→
India Derived Role
Comp Code: 100
+
USA Derived Role
Comp Code: 300
Master vs Derived — Key Difference
🔑
Only Org Level Values differ. Menu, auth objects, and auth field values are inherited from master.
  • No limit on derived roles per master role
  • Depends on number of branches/business units
Important Tables (AGR*)
  • AGR_DEFINE All role definitions & master/derived info
  • AGR_AGRS Single/Composite role info
  • AGR_TCODES Role and its T-codes
  • AGR_USERS Users assigned to roles
  • AGR_1251 Role and its Auth Values
  • AGR_1252 Role and its Org Values

09 — TROUBLESHOOTING

Common Authorization Issues

🚫 Issue 1: Cannot Login to SAP System ▾
  • Incorrect user password
  • User is locked
  • User account expired
  • Password expired (controlled by profile parameter)
  • Incorrect Client entered
⚠ Issue 2: Logged In but Cannot Execute a T-Code ▾
  • User does not have access to that T-Code
  • Role is expired
  • Profile not generated for the role
  • User Comparison not done
✅ Issue 3: Can Execute T-Code but Cannot Perform a Function Within It ▾
Example: Can open SU01 but cannot assign roles to users
  • Missing Auth Values for that specific function
  • Profile not regenerated after role change
  • User Comparison not done
10 — TOOLS

SU53 — Auth Check & SU56 — User Buffer

SU53
Evaluate Last Auth Check
  • Shows if last auth check was Success or Failure
  • On failure: shows the Auth Object, Auth Field, and required values
  • Can also run for other users: SU53 → Menu → Evaluate → Other User
SU56
User Buffer
  • Temporary memory created when user logs in
  • Wiped out when user logs out
  • System checks User Buffer (not DB) when executing T-codes
🔧
If buffer issue:
1) Ask user to logout and re-login
2) Ask user to reset buffer via SU56
11 — TRACE TOOLS

ST01 System Trace & STAUTHTRACE

ST01 — Return Codes
RC CodeMeaning
RC = 0 Authorization check SUCCESSFUL
RC = 4 User has the auth object, but different auth field values
RC = 8 User does not have the required auth values in User Buffer
RC = 12 User does NOT have the auth object at all
ST01 — Important Note
⚠
ST01 trace is applied to the local application server only. Make sure the user and you are on the same server before applying trace.
Find server info:
AL08 — Users in each server (server-wise)

SM51 — List of all application servers
STAUTHTRACE
Exclusive Authorization Trace
  • Exclusive auth trace — more powerful than ST01
  • Trace applied across ALL application servers
  • No need to match application servers
💡
0 Records Read Reasons: Incorrect input values, or the user is present on a different application server (use STAUTHTRACE to avoid this issue).
12 — SUIM

User Information System

🔍
SUIM — User Information System. Used to query and find information about users, roles, profiles, and authorizations across the system.
Common SUIM Queries
  • Find all users who have access to SU01 with Activity: 60
  • Find users whose ID starts with 'S' AND have access to SU01
  • Find all locked users
  • Find roles containing: SU01, SU10, PFCG, SU53, SU56
  • Find roles starting with 'X' that contain SU01
  • Find roles that give access to auth values: S_TABU_DIS → Activity: 03, Auth Group: SS
  • Find all Composite Roles in the system
  • Find T-codes assigned to user USERGRC14
  • Find all Auth Objects which contain Activity as a field
13 — PFUD

Mass User Comparison

🔄
PFUD performs user comparison for multiple roles at once. Program: PFCG_TIME_DEPENDENCY
Actions Performed in PFUD
Profile Matchup
  • Ensures profiles get assigned to users
  • Makes role status green under SU01
Composite Role Reconciliation
  • Reflects composite role changes to users
  • Syncs added/removed single roles
HR Org Mgmt Reconciliation
  • Ensures roles assigned to org positions reach users
Cleanups
  • Removes expired roles from SU01 records
