User Types — All 5
1️⃣ Dialog — Default User Type▾
  • Interactive user — humans logging in daily
  • Password parameters ARE applied
  • GUI Login is allowed
  • Example: All employees of the organization
2️⃣ Service — Multiple Dialog Logons▾
  • Used for multiple dialog logons simultaneously
  • Password parameters are NOT applied
  • GUI Login is allowed
  • Example: FFID in GRC, TEST IDs
3️⃣ System — Internal Communication▾
  • Password parameters are NOT applied
  • GUI Login NOT allowed
  • Example: Background jobs, Internal RFC
4️⃣ Communication — External Communication▾
  • Password parameters are NOT applied
  • GUI Login NOT allowed
  • Example: RFC (Remote Function Call)
5️⃣ Reference — Extra Access Provider▾
  • Used when Dialog user hits the 312-profile limit
  • No password required
  • Cannot login through GUI
  • Provides extra access to Dialog IDs

04 — AUTHORIZATION DATA

Auth Objects, Fields, Org Levels & Traffic Lights

🚦 Traffic Lights — Auth Object Status Indicators in PFCG
💡
When you open the Authorization tab in PFCG, each auth object is shown with a colour-coded indicator (Traffic Light) showing its fill status.
GREEN — All auth fields are filled / maintained with values.
Symbol: CO = Maintained authorization fields (fully green)
YELLOW — At least one auth field is NOT maintained (still empty).
Symbol: C&O = Partially unmaintained authorization fields
RED — Unmaintained Org Level — Org fields have not been filled.
Symbol: %CO = Unmaintained organizational levels
⚠
A role should ideally show all GREEN before being assigned to users. Yellow means incomplete auth data; Red means org levels are not filled.
Auth Object Status (Text Labels)
  • Standard SAP default values pulled from SU24 for the T-code added in Menu
  • Maintained Blank fields have been filled with values
  • Changed SAP default values have been modified
  • Manually Auth Object inserted directly into the Role (not pulled from SU24)
Normal Field vs Org Field
Normal FieldOrg Field
Does NOT represent organization
e.g., Activity, Role Name		 Represents organization
e.g., Plant, Cost Center, Company Code
Value meaning is universal
Activity 03 = Display everywhere		 Value meaning differs per org
BMW: Plant 100 = UK Plant
Walmart: Plant 100 = USA Branch
Maintained locally within auth Maintained centrally in Org Levels tab
Deleting Auth Objects from a Role (Prerequisites)
  • Auth Object must be Deactivated first
  • Auth Object status must be either MANUALLY or CHANGED
Profile Generation — 2 Modes
Manual Mode
Go to Change Authorization Data and click Generate Profile.
Used during role creation.
Expert Mode
Used during Role Modification ONLY (not creation). Has 3 sub-options.
Expert Mode — 3 Options (Role Modification Only)
🗑 Option 1: Delete and Recreate Profile and Authorizations ▾
  • Deletes entire old data including customization
  • Freshly loads auth data from SU24 for all T-codes in Menu
  • Use when: customization needs to be deleted (old data is lost)
✏ Option 2: Edit Old Status ▾
  • Keeps old data as-is (including customization)
  • Does NOT pull auth data for newly added T-codes in menu
  • Use when: auth data needs direct modification without changing Menu
🔀 Option 3: Read Old Status and Merge with New Data  ✓ RECOMMENDED when Menu changed ▾
  • Keeps old data as-is (including customization)
  • Also pulls auth data for ALL T-codes in Menu (including new ones)
  • Merges old data with new data wherever possible
  • Best option when role menu has been changed

06 — DERIVED ROLES

Master Role → Derived Roles

🌿
A Derived Role (Z) is a single role derived from another single role (Master Role). Only Org Level values differ. All menu & auth structure is inherited from the master.
When to Use Derived Roles?
🌐
When the client is an MNC (Multinational Company) with multiple business units — same T-codes but different org values (different Company Codes, Plants, Cost Centers).
Examples: Coke, BMW, NESTLE, TATA, DUPONT, TOTAL, UNILEVER
Example: India vs USA Finance
Both process invoices using T-code FBVO, linked to auth object F_FICO_BUK. Access differentiated by Company Code (Org Field).
Master Role
FBVO / F_FICO_BUK
Comp Code: * (all)
→
India Derived Role
Comp Code: 100
+
USA Derived Role
Comp Code: 300
Master vs Derived — Key Points
  • Difference: Only Org Values differ
  • Limit: No limit on derived roles per master
  • Depends on how many branches/business units the company has
  • To find master of a derived role → check AGR_DEFINE
Important Tables — AGR* (Activity Group)
💡
AGR = Activity Group = Role Name. All role-related tables start with AGR*.
  • AGR_DEFINE All role definitions & master/derived info
  • AGR_AGRS Single & Composite role info
  • AGR_TCODES Role and its T-codes
  • AGR_USERS Users assigned to roles
  • AGR_1251 Role and its Auth Values
  • AGR_1252 Role and its Org Values

09 — TROUBLESHOOTING

Common Authorization Issues

🚫 Issue 1: Cannot Login to SAP System ▾
  • Incorrect user password
  • User is locked
  • User account expired
  • Password expired (controlled by profile parameter)
  • Incorrect Client entered at login screen
⚠ Issue 2: Logged In but Cannot Execute a T-Code ▾
  • User does not have access to that T-Code
  • Role is expired
  • Profile not generated for the role
  • User Comparison not done
✅ Issue 3: Can Execute T-Code but Cannot Perform a Specific Function Within It ▾
Example: Can open SU01 but cannot assign roles to users
  • User does not have the appropriate Auth Values for role assignment
  • Profile not regenerated after role change
  • User Comparison not done
T-Codes Used for Troubleshooting
SU53 SU56 ST01 STAUTHTRACE
10 — TOOLS

SU53 — Auth Check & SU56 — User Buffer

SU53
Evaluate Last Auth Check
  • Shows if last auth check was Success or Failure
  • On failure: shows the Auth Object, Auth Field, and required values
  • Run for other users: SU53 → Menu → Evaluate → Other User
SU56
User Buffer
  • Temporary memory created when user logs in
  • Wiped out when user logs out
  • System checks User Buffer (not DB directly) when executing T-codes
🔧
If buffer issue:
1) Ask user to logout and re-login
2) Ask user to reset buffer via SU56
11 — TRACE TOOLS

ST01 System Trace & STAUTHTRACE

ST01 — Return Codes (RC)
RC CodeMeaning
RC = 0Authorization check SUCCESSFUL
RC = 4User has the auth object, but different auth field values
RC = 8User does not have required auth values in User Buffer
RC = 12User does NOT have the auth object at all
ST01 — Critical Note
⚠
ST01 trace is applied to the local application server only. Ensure the user and you are on the same server before applying trace.
0 Records Read? Could mean:
  • Incorrect input values entered
  • User is present on a different application server
Find server info:
AL08 — Users in each server (server-wise)

SM51 — List of all application servers
STAUTHTRACE
Exclusive Authorization Trace
  • More powerful than ST01
  • Trace applied across ALL application servers
  • No need to worry about matching application servers
  • Exclusive Authorization Trace — focused on auth checks only
12 — SUIM

User Information System

🔍
SUIM — User Information System. Used to query and find information about users, roles, profiles, and authorizations across the system.
Common SUIM Queries (Practice)
  • Find all users who have access to SU01 with Activity: 60
  • Find users whose User ID starts with 'S' AND have access to SU01
  • Find all locked users in the system
  • Find roles containing T-codes: SU01, SU10, PFCG, SU53, SU56
  • Find roles starting with 'X' that contain SU01 T-code
  • Find roles that give access to auth values: S_TABU_DIS → Activity: 03, Auth Group: SS
  • Find complete list of all Composite Roles in the system
  • Find T-codes assigned to user USERGRC14
  • Find all Auth Objects which contain Activity as a field
13 — PFUD

Mass User Comparison

🔄
PFUD performs user comparison for multiple roles at once.
Background program: PFCG_TIME_DEPENDENCY
💡
When there is a requirement to perform user comparison for 100 roles (multiple roles), use T-code PFUD instead of doing it role by role in PFCG.
Actions Performed in PFUD
1. Profile Matchup
  • Ensures profiles get assigned to users
  • Makes role status green under SU01
2. Composite Role Reconciliation
  • When composite role is modified (role added/removed), reflects change to users
3. HR Org Mgmt Reconciliation
  • Ensures roles assigned to org positions/org units reach the users under those positions
4. Cleanups
  • Removes expired roles from SU01 user records
