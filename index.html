00C — FOUNDATIONS

Client Administration

🖥
A Client is an independent space within a SAP system, represented by a 3-digit number (000–999).
Default Clients — Non-IDES System
000001006
Default Clients — IDES System
000001066800810811812
IDES = International Demonstration and Education System
Client Data Types
Client Independent Data
Standard Data — same across all clients
  • T-codes
  • ABAP Code / Programs
  • Repository objects
Client Dependent Data
Customized Data — specific per client
  • Users
  • Transactional Data
  • Business Data / Config
Client T-Codes (SCC*)
T-CodeDescription
SCC4Client Creation / client settings maintenance
SCC5Client Deletion
SCCLLocal Client Copy
SCC9Remote Client Copy
SCC4 / Client Administration Related Tables
TableWhy It Matters
T000Main client table. SCC4 client settings and protections revolve around this table.
TSTCTechnical transaction definition table. Useful when checking transaction metadata for SCC4/SCCL/SCC9.
TSTCTTransaction description texts for client administration transactions.
TDDATTable authorization groups. Relevant when protecting sensitive tables through S_TABU_DIS.
USR02User logon data by client. Helpful when checking user impact in a specific client.
⚠
T000 is the key table to remember for SCC4. In practice, access to it is typically protected very tightly.
00D — FOUNDATIONS

User Administration — SU01

👤
All user-related activities are handled via SU01 — User Maintenance.
User Activities in SU01
User CreationUser ModificationUser DeletionUser Lock / UnlockUser CopyPassword Reset
SU01 Tabs — All Explained
TabPurpose
AddressFirst name, last name, email, department
Logon DataUser Type, User Group, Validity Period
SNCSecure Network Communication — when activated, no password needed to login
DefaultsDate format, decimal notation, time format, printer details
ParametersOnly entered when requested by user (pre-fills field defaults)
RolesAssign authorizations through roles
ProfilesAuto-assigned once roles are assigned (interlinked with roles)
GroupsAssign user to extra groups if they belong to multiple groups
PersonalizationDo not enter any details here
License DataRelated to user licensing
Change DocsView all changes to a user in a time interval
Key Facts to Remember
  • Max User ID length: 12 characters
  • Naming: First Name + Last Name
  • Max profiles per user: 312
  • Mandatory fields: Last Name + Password
  • User tables start with: USR*
  • USR02 — User Logon Data
  • USR40 — Illegal Passwords list
01 — PROFILE CONCEPT

SAP Versions & Profile History

Version History
3.0 → 4.6a → 4.6b → 4.6C → 4.7 → ECC 5 → ECC 6.0 + EHP1..8 ECC = ERP Central Component EHP = Enhancement Package
Key Version Milestone
⚠
Version 4.7 — major changes in Security area. The role concept was introduced from 4.7 onwards. Prior versions used the profile concept via SU02.
Types of Profiles
Standard Profiles
  • Come with SAP installation
  • Can be assigned to users directly
  • SAP recommends never assigning them directly
Generated Profiles
  • Created through roles in PFCG
  • Cannot be assigned to users directly
  • Usually start with T
  • Assign the role instead
02 — ROLE ADMINISTRATION

Role Basics & Role Matrix

What is a Role?
📦
A Role is a group of T-codes and the related authorizations. Each team in a project usually gets its own set of roles.
Role Categories
  • Technical / Support Roles — for support users
  • Functional / Business Roles — for end users
Role Matrix
  • Information gathered from all technical teams
  • Lists roles needed per team and access type
  • Defines the access level required per role
03 — PFCG

Profile Generator — Role Types & Naming

ℹ
PFCG = Profile Generator, the main T-code used for role creation and management.
X
Single Role
One set of T-codes and authorizations
Y
Composite Role
Group of single roles bundled together
Z
Derived Role
Inherited from a master single role
Role Naming Convention (Max 30 Characters)
[Role Type] : [Business Process] : [Sub Process] : [Extra Info] X:HR:PERSONNEL_ADMIN:BMW X:BS:PROGRAM_EXECUTE:MURALI1
04 — AUTHORIZATION DATA

Auth Objects, Fields, Org Levels & Traffic Lights

Traffic Lights — Auth Object Status in PFCG
GREEN — all auth fields are filled or maintained.
YELLOW — at least one auth field is not maintained.
RED — org level fields are not maintained.

Master roles eliminate duplication by creating one versioned role that multiple derived roles can inherit from.

06 — DERIVED ROLES

Master Role → Derived Roles

Important Tables (AGR*)
  • AGR_DEFINE role definitions and master/derived info
  • AGR_AGRS single and composite role info
  • AGR_TCODES role and its T-codes

07 — ROLE MODIFICATION

Modifying Single, Composite & Derived Roles

Single Role
1
Modify menu and T-codes
2
Modify auth data directly if needed
Composite Role
  • Add or remove single roles
  • Single roles are still maintained separately
Derived Role
⚠
Only org-level values should differ. Menu and auth structure come from the master role.

10 — TOOLS

SU53 — Auth Check & SU56 — User Buffer

SU53
Evaluate Last Auth Check
  • Shows whether the last auth check was success or failure
  • On failure it shows object, field, and required values
SU56
User Buffer
  • Temporary authorization memory created when user logs in
  • Cleared when user logs out
  • System checks the user buffer when executing T-codes
11 — TRACE TOOLS

ST01 System Trace & STAUTHTRACE

ST01 — Return Codes (RC)
RC CodeMeaning
RC = 0Authorization check successful
RC = 4Object exists but field values differ
RC = 8Required auth values not found in user buffer
RC = 12User does not have the object
